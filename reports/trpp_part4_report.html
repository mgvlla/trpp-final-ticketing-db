Итоговый проект по дисциплине ТРПП

Часть 4. Подготовка к защите проекта
Выполнили: Магомедова Лолита, Багирян Марьям

1. Краткое содержание проекта для защиты

Проект посвящен разработке базы данных для сервиса продажи билетов на мероприятия. В базе данных реализовано хранение сведений о категориях мероприятий, организаторах, площадках, пользователях, билетах, заказах и оплатах. Реализация выполнена в PostgreSQL, а запуск осуществляется в Docker, что делает проект переносимым и удобным для демонстрации.

3. Пример устного выступления

В рамках итогового проекта нами была разработана реляционная база данных для сервиса продажи билетов на мероприятия. Цель проекта состояла в том, чтобы спроектировать структуру хранения данных о мероприятиях, клиентах, билетах, заказах и платежах. На этапе проектирования были выделены основные сущности и связи между ними, после чего схема была реализована в PostgreSQL. Для упрощения запуска проекта использован Docker. Также были подготовлены тестовые данные и демонстрационные SQL-запросы, позволяющие анализировать продажи и наличие билетов. Таким образом, полученное решение может использоваться как основа для дальнейшей разработки полноценного информационного сервиса.

4. Возможные вопросы и ответы

ВопросКраткий ответ
Почему выбрана PostgreSQL?Потому что это популярная реляционная СУБД с удобной поддержкой SQL и хорошими средствами контроля целостности данных.
Почему использован Docker?Docker упрощает запуск проекта на любом компьютере и позволяет не устанавливать PostgreSQL вручную.
Какие связи реализованы в базе?В основном связи типа «один ко многим»: категория и мероприятия, пользователь и заказы, заказ и строки заказа.
Зачем нужна нормализация?Нормализация уменьшает избыточность данных и снижает риск появления противоречий при обновлении записей.
Как можно развить проект дальше?Можно добавить роли пользователей, возвраты билетов, историю изменений и веб-интерфейс.
