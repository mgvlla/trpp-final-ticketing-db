Часть 4. Подготовка к защите проекта
Выполнили: Магомедова Лолита, Багирян Марьям, Гаврилова Софья

1. Краткое содержание проекта для защиты

Проект посвящён разработке реляционной базы данных для сервиса продажи билетов на мероприятия. В базе данных реализовано хранение сведений о категориях мероприятий, организаторах, площадках, пользователях, билетах, заказах и оплатах. Схема базы данных нормализована до третьей нормальной формы (3НФ), содержит 12 связанных таблиц и обеспечивает целостность данных на уровне ограничений (CHECK, UNIQUE, NOT NULL, ON DELETE CASCADE). Реализация выполнена в PostgreSQL 16, запуск осуществляется в Docker. Дополнительно настроена система сборки (requirements.txt, setup.py) и документации (Sphinx). Проект опубликован на GitHub.

3. Пример устного выступления

Слайд 1 (Лолита): Здравствуйте! Меня зовут Магомедова Лолита. Со мной в команде Багирян Марьям и Гаврилова Софья. Представляем итоговый проект по дисциплине «Технологии разработки программных приложений». Тема — «Разработка реляционной базы данных для продажи билетов на мероприятия».

Слайд 2 (Софья): Идея проекта возникла из наблюдения за работой реальных сервисов — Яндекс Афиша, Кассир.ру, Timepad. Многие небольшие организаторы до сих пор ведут учёт в Excel, что приводит к путанице с местами и ошибкам в отчётности.

Слайд 4 (Лолита): Решение состоит из четырёх компонентов: macOS, Docker Desktop, PostgreSQL 16, SQL. PostgreSQL выбрана за поддержку ограничений целостности. Docker обеспечивает воспроизводимость.

Слайд 5 (Марьям): База данных поддерживает пять ключевых функций: хранение мероприятий, управление билетами, оформление заказов, фиксация оплат, аналитические запросы.

Слайд 7 (Софья): Аналоги: Яндекс Афиша, Кассир.ру, Timepad, Ticketmaster. Это закрытые системы. Наши особенности: открытый исходный код, Docker-инициализация, учебная наглядность, SQL-запросы для аналитики.

Слайд 11 (Демонстрация): Лолита вводит команды в терминале: docker compose down -v, docker compose up -d, docker ps, psql, \dt, SELECT * FROM event, SELECT * FROM ticket, 03_queries.sql. Марьям комментирует результаты. Софья подводит итог.

Слайд 12 (Софья): Результаты: 12 нормализованных таблиц, Docker, тестовые данные, 5 запросов, код на GitHub. Возможности развития: веб-интерфейс, возвраты, роли, история, онлайн-оплата.

Слайд 13 (Лолита): Цель проекта достигнута: спроектирована и реализована реляционная база данных для сервиса продажи билетов на мероприятия. Спасибо за внимание.

4. Возможные вопросы и ответы

ВопросОтветКто отвечает
Почему выбрана PostgreSQL?Это реляционная СУБД с открытым исходным кодом, поддержкой ограничений целостности и транзакционностью.Лолита
Почему использован Docker?Docker обеспечивает воспроизводимость окружения. Проект запускается одной командой без установки PostgreSQL вручную.Лолита
Почему Docker Compose, а не отдельный Dockerfile?В проекте используется и то, и другое. Dockerfile упаковывает приложение, Docker Compose упрощает локальный запуск.Лолита
Что такое 3НФ и зачем она нужна?3НФ исключает дублирование данных. Например, статус билета вынесен в отдельную таблицу.Марьям
Какие связи реализованы в базе?Связи «один ко многим»: категория — мероприятия, пользователь — заказы, заказ — позиции заказа.Марьям
Зачем setup.py и requirements.txt?Формальное соответствие методическим указаниям и задел на будущее для Python-интеграции.Марьям
Как настроен Sphinx?Конфигурация в docs/source/conf.py, сборка командой sphinx-build, результат в docs/build/.Марьям
Какие аналоги есть и в чём особенности?Яндекс Афиша, Кассир.ру, Timepad — закрытые системы. Наши особенности: открытый код, Docker, SQL-аналитика.Софья
Какие тестовые данные заложены?3 категории, 2 организатора, 2 площадки, 3 пользователя, 3 мероприятия, 7 билетов, 2 оплаты.Софья
Как распределялась работа в команде?Лолита — структура БД и Docker. Марьям — тестовые данные, запросы, Sphinx. Софья — документация, презентация, анализ аналогов.Софья
Как можно развить проект?Веб-интерфейс, возвраты билетов, роли пользователей, история операций, онлайн-оплата.Марьям
