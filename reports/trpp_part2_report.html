Часть 2. Проектирование базы данных
Выполнили: Магомедова Лолита, Багирян Марьям

1. Логическая модель

На этапе логического проектирования были выделены основные сущности и установлены связи между ними. Таблица event связана с таблицами event_category, organizer и venue, поскольку каждое мероприятие имеет категорию, организатора и площадку. Таблица ticket хранит билеты, относящиеся к конкретному мероприятию. Заказы пользователей представлены таблицами customer_order и order_item, а сведения об оплате вынесены в таблицу payment.

3. Связи между таблицами

Между таблицами установлены связи типа «один ко многим». Одна категория может соответствовать многим мероприятиям, один организатор может проводить несколько мероприятий, а одна площадка может использоваться для множества событий. Один пользователь может сформировать несколько заказов, но каждый заказ принадлежит только одному пользователю. Один заказ может включать несколько строк заказа, каждая из которых соответствует одному билету. Такая структура позволяет избежать дублирования данных и соответствует нормализованной реляционной модели.

4. Нормализация

Схема приведена к третьей нормальной форме. Каждая таблица хранит данные только об одном объекте предметной области, составные данные вынесены в отдельные сущности, а справочные значения представлены отдельными таблицами. Благодаря этому уменьшается избыточность хранения и упрощается поддержка целостности данных.

5. DDL-фрагмент

Ниже приведен пример создания одной из ключевых таблиц:

CREATE TABLE event ( event_id SERIAL PRIMARY KEY, event_name VARCHAR(150) NOT NULL, event_date TIMESTAMP NOT NULL, base_price NUMERIC(10, 2) NOT NULL CHECK (base_price >= 0), description TEXT, event_category_id INTEGER NOT NULL REFERENCES event_category(event_category_id), organizer_id INTEGER NOT NULL REFERENCES organizer(organizer_id), venue_id INTEGER NOT NULL REFERENCES venue(venue_id) );

6. Вывод

Во второй части проекта была разработана логическая структура базы данных, определены таблицы, атрибуты и связи между ними. Полученная модель пригодна для последующей реализации в PostgreSQL и выполнения прикладных запросов для анализа продаж билетов и мероприятий.
